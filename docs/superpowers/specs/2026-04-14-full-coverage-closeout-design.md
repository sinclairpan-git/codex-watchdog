# 全量未落地清零执行设计（基于覆盖性审计矩阵）

## 目标

清空覆盖性审计矩阵中全部未落地条款（当前 115 行），并保证每条条款在“实现 / 验证 / 入口”三证齐备后才允许移出矩阵。

## 执行范围与批次策略

1. **执行范围**：仅针对 `docs/superpowers/specs/2026-04-14-coverage-audit-matrix.md` 中的“未落地”行（不含非执行性条款）。
2. **排序原则**：按 `WI` 编号从小到大推进，每批最多包含 1 个主 WI + 若干“直接依赖”的共享能力。直接依赖指：若不补该能力，本批至少一条目标条款无法同时获得实现/验证/入口三证。
3. **共享能力准入条件**：
   - 必须是当前批次条款闭环的直接阻塞项；
   - 若仅服务 1 条目标条款，需在批次闭环清单中说明为何不并批处理；
   - 不得引入与本批无关的新接口、抽象层或目录重组。
4. **批次规模**：按依赖闭包成批，行数仅作上限参考；本批引入的共享能力若用于支撑本批条款关闭，其证据必须在本批内闭环，不得依赖下一批补证据后再回溯关闭本批条款。
5. **批次完成定义（硬门槛）**：
   - 本批目标条款全部具备实现/验证/入口三类证据；
   - 矩阵已更新并重算统计；
   - `latest-summary` 已同步更新且列出证据位点；
   - 批次闭环清单已生成（含最小回归结果留档）。
   - 若本批仅关闭部分矩阵行，其余受影响但未关闭条目必须在闭环清单中显式列出。
   未满足任何一项则该批不得关闭。

## 证据闭环与矩阵对账

1. **移出矩阵的唯一条件**：条款同时具备“实现证据 + 验证证据 + 正式入口证据”三者齐备。
2. **证据准入规则**：
   - 实现证据：必须是代码/配置/可运行模块；文档与总结只能作为索引，不可替代实现证据。
   - 验证证据：必须是测试、验收脚本或可执行检查；仅有 summary 不可视为验证。
   - 入口证据：必须是主链路可触达的 API/CLI/运行时路由；示例脚本、预览入口、测试桩不算入口。行号必须对应实际证据落点，不得仅指向文件头。
3. **矩阵更新纪律**：
   - 每批只改本批涉及的矩阵行；
   - 每批更新后必须重算“未落地总数”和“缺口类型分布”，优先脚本化并将重算结果附到批次闭环清单；
   - 主 WI 的 `latest-summary` 必须同步更新；若本批修改了其他 WI 的交付边界或证据，也必须同步更新对应 WI 的 `latest-summary`。

## 共享能力与回归边界

1. **影响面识别**：共享能力变更必须附带影响清单，列明影响到的矩阵条目与最小回归集合。
2. **强制回归规则**：共享能力变更后必须执行最小回归集合；未完成回归的条目一律不得移出矩阵。
   - 最小回归集合依据：入口变化、共享模块变更、受影响矩阵条目三类生成。
3. **失败即回退**：回归失败或主链路验证失败时，相关矩阵行必须保持“未落地”或恢复为“未落地”，不得保留已关闭状态。

## 批次闭环清单（必须产物）

每批必须输出以下清单，缺一不可：

- 涉及 `WI`
- 涉及矩阵条目
- 新增实现证据（文件路径 + 行号）
- 新增验证证据（测试/脚本路径 + 行号）
- 新增入口证据（API/CLI/路由路径 + 行号）
- 回归范围 / 回归结果 / 失败处置
- 仍未闭环原因（若有）
- 涉及缺口类型变化

## 失败态处理

1. 共享能力回归失败：涉及条款全部保留为未落地，并在 `latest-summary` 中记录阻塞原因。
2. 证据不齐：不得提前清矩阵，不得以“计划补齐”替代实际证据。
3. 主链路验证失败：本批直接判为未完成，重新修复后再进入矩阵更新。
4. 若矩阵已更新后才发现回归失败，必须在同一次修正中恢复统计与对应条目状态。

## 归属规则

1. 每条未落地矩阵行在进入执行前必须绑定唯一主 WI。
2. 若无现成 WI 可承接，先补建或指定承接 WI，再进入批次排序与实施。
